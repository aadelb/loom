TECHNICAL INTEGRATION PATTERNS
================================================================================

1. TOR NETWORK ACCESS
   - Use existing TOR_ENABLED environment variable
   - Leverage tor.py module for Tor client management
   - Handle extended timeouts (Tor is slower than clearnet)
   - Expect 30-60 second response times

2. URL VALIDATION & SECURITY
   - All .onion URLs must pass validators.validate_url()
   - Prevent SSRF attacks via domain validation
   - Whitelist only .onion and known infrastructure
   - Sanitize all output

3. CACHING STRATEGY
   - Dark web content changes less frequently than clearnet
   - Use content-hash cache with 24-48 hour TTL
   - Implement optional force-refresh parameter
   - Cache at multiple levels (tool output, intermediate results)

4. ERROR HANDLING
   - .onion services frequently offline (expect failures)
   - Tor circuit failures should trigger exponential backoff retry
   - Timeouts are normal (extend standard timeout to 60+ seconds)
   - Return partial results when possible

5. RESULT NORMALIZATION
   - Standardize output schema across all dark web tools
   - Include: source, timestamp, confidence_score, threat_level
   - Implement deduplication for multi-tool results
   - Sort by relevance/threat score

================================================================================

SECURITY CONSIDERATIONS
================================================================================

1. Tor client must be running locally
2. Rate limiting essential (dark web operators monitor for scanning)
3. Respect robots.txt and service Terms of Service
4. Log all queries (audit trail + threat intelligence)
5. Consider residential proxies for sensitive queries
6. Implement request fingerprinting to avoid detection
7. Randomize User-Agent headers
8. Add delays between requests to appear human
