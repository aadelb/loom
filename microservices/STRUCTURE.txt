CONFIGURATION
================================================================================

Service Configuration Hierarchy (in order of precedence):
  1. Environment variables (LOOM_SERVICE, LOOM_PORT, etc.)
  2. Service-specific config.json (if exists at /etc/loom/*/config.json)
  3. Compiled defaults (from src/loom/config.py)

Key Environment Variables:
  ├── LOOM_SERVICE: Service type (core|redteam|intel|infra)
  ├── LOOM_PORT: Port number (8787, 8788, 8789, 8790)
  ├── REDIS_URL: Redis connection string
  ├── DATABASE_URL: PostgreSQL connection string
  ├── LOG_LEVEL: INFO|DEBUG|WARNING|ERROR
  ├── LLM_CASCADE_ORDER: Provider fallback order
  ├── ENABLE_*_TOOLS: Per-service tool enabling
  └── TOR_ENABLED: Enable Tor network integration (loom-intel only)

Feature Flags (enable/disable tool categories):
  ├── ENABLE_FETCH_TOOLS (core)
  ├── ENABLE_SEARCH_TOOLS (core)
  ├── ENABLE_REFRAME_TOOLS (redteam)
  ├── ENABLE_SAFETY_TOOLS (redteam)
  ├── ENABLE_DARK_WEB_TOOLS (intel)
  ├── ENABLE_THREAT_INTEL_TOOLS (intel)
  ├── ENABLE_BILLING_TOOLS (infra)
  ├── ENABLE_AUTH_TOOLS (infra)
  ├── ENABLE_JOB_TOOLS (infra)
  └── ENABLE_MONITORING_TOOLS (infra)

================================================================================
SCALING RECOMMENDATIONS
================================================================================

Horizontal Scaling (add more replicas):
  ✓ loom-core: Stateless, can run 3-5 replicas
  ✓ loom-intel: Stateless, can run 2-4 replicas
  ~ loom-infra: Mostly stateless, 2 replicas recommended
  ~ loom-redteam: Shared strategy registry, 2 replicas max

Vertical Scaling (add more resources):
  • loom-core: Increase concurrency (SPIDER_CONCURRENCY, LLM_MAX_PARALLEL)
  • loom-redteam: Increase memory & CPU (LLM reasoning is compute-intensive)
  • loom-intel: Increase network bandwidth (Tor, dark web queries)
  • loom-infra: Already low-resource

Database Scaling:
  PostgreSQL: Use connection pooling (PgBouncer), streaming replication
  Redis: Use Redis Cluster or Sentinel for HA
